# Sidebar Module — How It Works

> 사람용. 설계 철학과 동작 원리.  
> 코드 상세는 → [manual-detail-sidebar.md](./manual-detail-sidebar.md)

## 사이드바가 하는 일

사이드바는 사용자의 마크다운 폴더들을 관리하는 공간이다. "즐겨찾기" 개념으로 원하는 폴더를 등록하고, 그 안의 파일들을 탐색/정렬/이동/삭제한다.

```
┌─────────────────────────────┐
│  Documents                  │
│  ─────────────────────────  │
│  📁 연구 노트 (12)      + + │
│  ├── 실험 기록.md            │
│  ├── 논문 정리.md            │
│  └── 📁 참고자료              │
│  ──────────────────────────  │
│  📁 회의록 (5)           + + │
│  ├── 2024-04 주간.md         │
│  └── ...                     │
│  ─────────────────────────── │
│  [접기] [컴팩트] [정렬] [추가] │
└─────────────────────────────┘
```

## 핵심 설계 결정

### 왜 "즐겨찾기 폴더" 방식인가

많은 노트 앱이 자체 데이터베이스나 전용 디렉토리를 쓴다. Marknote는 기존 파일 시스템의 아무 폴더나 즐겨찾기로 등록하는 방식을 택했다.

**장점**: 
- OneDrive, Google Drive 등 클라우드 동기화 폴더를 그대로 쓸 수 있다
- 다른 앱(VS Code, Obsidian 등)과 같은 폴더를 공유할 수 있다
- 앱을 삭제해도 파일은 그대로 남는다

**트레이드오프**:
- 폴더가 외부에서 변경될 수 있어 FS 감시(watch)가 필요하다
- 즐겨찾기 폴더가 이동/삭제되면 "깨진 경로"를 처리해야 한다

### FS 감시 (실시간 동기화)

사용자가 탐색기에서 파일을 추가/삭제/이름변경하면, 앱이 이를 바로 반영해야 한다. `tauri-plugin-fs`의 `watch()` API로 모든 즐겨찾기 폴더를 재귀적으로 감시한다.

```
watch(favorites, { recursive: true, delayMs: 500 })
  → 변경 감지 → 500ms 디바운스 → refreshFileTree()
```

500ms 디바운스는 파일 복사 같은 대량 작업에서 불필요한 갱신을 줄인다.

## 파일 트리 — 탐색과 관리

### 정렬

3가지 정렬 모드가 있고, 모든 모드에서 폴더는 항상 위에 온다:

| 모드 | 규칙 |
|------|------|
| 이름순 | `localeCompare` (한국어 자연정렬) |
| 날짜순 | 수정일 내림차순 (최신 먼저) |
| 커스텀 | 사용자가 드래그로 정한 순서. persist 됨 |

커스텀 정렬은 같은 폴더 안에서 파일을 드래그하면 자동으로 활성화된다.

### 멀티 선택

파일 관리에서 흔히 쓰는 패턴을 그대로 구현했다:

- **일반 클릭**: 단일 선택
- **Ctrl+클릭**: 토글 (기존 선택 유지하면서 추가/해제)
- **Shift+클릭**: 범위 선택 (마지막 클릭~현재 클릭 사이 전부)

선택된 항목들은 한꺼번에 삭제하거나 다른 폴더로 드래그 이동할 수 있다.

### 드래그 이동

```
같은 폴더에서 드래그 → 순서 변경 (커스텀 정렬)
다른 폴더로 드래그   → 파일/폴더 이동
```

이동 시 자동으로 처리되는 것들:
- 중복 이름이면 `(1)`, `(2)` 자동 추가
- 열려있는 탭의 경로 업데이트
- 즐겨찾기 파일 경로 업데이트
- 마크다운 문서면 연결된 이미지 경로도 업데이트
- 폴더 펼침 상태 유지

모든 이동은 Ctrl+Z로 되돌릴 수 있다.

## Undo/Redo — 파일 작업도 되돌리기

에디터의 Ctrl+Z는 텍스트 편집을 되돌리지만, 에디터 밖에서 Ctrl+Z를 누르면 파일 작업을 되돌린다.

되돌릴 수 있는 작업:
- 파일/폴더 이동
- 파일/폴더 삭제 (`.trash/`에서 복원)
- 이름 변경 (이미지 경로까지 복원)
- 즐겨찾기 폴더 순서 변경
- 파일 순서 변경 (커스텀 정렬)

**구현**: Command 패턴. 각 작업이 `execute()`와 `undo()` 함수를 가진다. 최대 50개 히스토리. 에디터 포커스가 아닌 상태에서 Ctrl+Z → undo, Ctrl+Shift+Z → redo.

## 앱 내 휴지통

파일을 삭제하면 즉시 영구 삭제하지 않고 `.trash/` 폴더로 옮긴다.

```
삭제 → .trash/{filename}-{timestamp}
Undo → .trash/ → 원래 위치 복원
앱 종료 → .trash/ 내용 → OS 휴지통

즐겨찾기 폴더마다 독립적인 .trash/ 폴더
```

**왜 OS 휴지통으로 바로 안 보내나**: OS 휴지통에서는 프로그래밍적으로 특정 파일을 복원하기 어렵다. 앱 내 `.trash/`를 쓰면 즉시 undo가 가능하고, 앱 종료 시 OS 휴지통으로 최종 이동해서 사용자가 나중에 복구할 수 있다.

## 슬라이딩 하이라이트

파일 트리에서 마우스를 움직이면 하이라이트가 부드럽게 따라다닌다. 호버된 항목의 위치를 측정해서 하이라이트 요소의 `top`/`height`를 CSS transition으로 변경한다.

이 패턴은 앱 전체에서 일관되게 사용된다 (사이드바, 탭바, 검색 탭 등).

## 커스터마이징

| 기능 | 방법 |
|------|------|
| 폴더 별명 | 실제 이름 대신 표시할 이름 설정 |
| 폴더 아이콘 | Lucide 아이콘 중 선택 |
| 사이드바 폭 | 오른쪽 가장자리 드래그 (180~500px) |
| 컴팩트 모드 | 줄 높이를 줄여 더 많은 항목 표시 |
| 폴더 순서 | 드래그로 즐겨찾기 폴더 순서 변경 |

모든 설정은 localStorage에 자동 저장되어 앱을 다시 열어도 유지된다.
