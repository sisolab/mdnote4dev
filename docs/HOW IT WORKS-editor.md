# Editor Module — How It Works

> 사람용. 설계 철학과 동작 원리.  
> 코드 상세는 → [manual-detail-editor.md](./manual-detail-editor.md)

## 에디터가 하는 일

마크다운 파일을 열면 WYSIWYG 편집기로 보여주고, 저장하면 다시 표준 마크다운으로 돌려놓는다. 중간에 이미지 붙여넣기, 파일 첨부, 표 편집 같은 리치 편집 기능을 제공한다.

```
┌─ 로드 ─────────────────────────────────────┐
│  .md 파일                                    │
│    → frontmatter 분리                        │
│    → @tiptap/markdown 파서 (marked)          │
│    → ProseMirror 문서 트리                    │
│    → .assets/ 링크 → fileAttachment 노드 변환 │
└──────────────────────────────────────────────┘

┌─ 편집 ─────────────────────────────────────┐
│  TipTap (ProseMirror) WYSIWYG               │
│    + 이미지 붙여넣기/삭제/undo               │
│    + 파일 첨부 카드                          │
│    + 코드블록 구문 강조                      │
│    + 표 편집                                 │
└──────────────────────────────────────────────┘

┌─ 저장 ─────────────────────────────────────┐
│  ProseMirror 문서 트리                       │
│    → editor.getMarkdown()                    │
│    → fileAttachment renderMarkdown           │
│    → frontmatter 복원                        │
│    → .md 파일 쓰기                           │
└──────────────────────────────────────────────┘
```

## 핵심 설계 결정

### 왜 @tiptap/markdown인가

처음에는 직접 만든 마크다운 파서(regex 기반, 366줄)를 썼다. 기본적인 텍스트/제목/리스트는 됐지만, 중첩 리스트, 코드블록 안의 특수문자, 번호 리스트 이어붙이기 등 엣지 케이스가 끝없이 나왔다. @tiptap/markdown 공식 확장으로 교체하면서 이 문제들이 한꺼번에 해결됐다.

**트레이드오프**: 공식 확장은 커스텀 노드(fileAttachment 등)를 자동으로 처리하지 못한다. `renderMarkdown` 훅을 수동으로 추가해야 하고, 로드 시에도 별도의 후변환 로직이 필요하다.

### 왜 frontmatter를 에디터 밖에서 관리하나

frontmatter(`---\ntags: [...]\n---`)는 YAML이지 마크다운이 아니다. TipTap에게 frontmatter를 넘기면 일반 텍스트로 렌더링되거나 파싱이 깨진다. 그래서:

1. **로드 시**: `stripFrontmatter()`로 제거한 본문만 에디터에 넣음
2. **저장 시**: `parseFrontmatter()`로 원본 frontmatter를 보존, 본문과 재결합

이렇게 하면 태그 같은 메타데이터는 상태바에서 별도로 편집하고, 에디터는 순수 콘텐츠만 다룬다.

## 이미지 붙여넣기 — 왜 이렇게 복잡한가

클립보드에서 이미지를 붙여넣는 건 단순해 보이지만, 실제로는 여러 단계가 필요하다:

```
클립보드 → blob 추출 → 파일 저장 → URL 변환 → 에디터 삽입 → 크기 결정
```

**복합 클립보드 문제**: 엑셀에서 복사하면 클립보드에 HTML과 이미지가 동시에 들어있다. HTML이 있으면 이미지 붙여넣기를 건너뛴다 (엑셀 표 대신 스크린샷이 붙는 걸 방지).

**파일 경로 → URL 변환**: Tauri 앱에서 로컬 파일을 `<img src>`로 보여주려면 `convertFileSrc()`로 `asset://localhost/...` URL을 만들어야 한다. 일반 `file://` 경로는 보안상 차단된다.

**재시도 로직**: 파일을 디스크에 쓴 직후 `<img>`로 로드하면 가끔 실패한다 (디스크 I/O 지연). 200ms 간격으로 2회 재시도한다.

**초기 크기**: 원본이 320px보다 작으면 원본 크기, 아니면 320px 기본. 사용자가 이미지 툴바로 크기를 조절할 수 있다.

## 에셋 라이프사이클 — 삭제와 되돌리기

이미지나 첨부파일을 에디터에서 삭제하면, 실제 파일을 즉시 지울 수 없다. Ctrl+Z로 되돌릴 수 있어야 하기 때문이다.

```
┌─ 삭제 ────────────────────────┐
│  에디터에서 이미지 삭제        │
│    → .assets/ → .trash/ 이동  │
│    → globalTrashMap에 기록     │
└───────────────────────────────┘

┌─ Undo ────────────────────────┐
│  Ctrl+Z                       │
│    → globalTrashMap에서 찾기   │
│    → .trash/ → .assets/ 복원  │
│    → img src 캐시 무효화       │
└───────────────────────────────┘

┌─ 앱 종료 ─────────────────────┐
│  .trash/ 내용 → OS 휴지통     │
│  (사용자가 나중에 복구 가능)   │
└───────────────────────────────┘
```

**globalTrashMap**: 모듈 레벨 `Map<filename, trashPath>`. 탭을 전환해도 유지된다. 에디터 `update` 이벤트에서 이전/현재 에셋 목록을 비교하고, 사라진 에셋을 `.trash/`로 옮기면서 Map에 기록한다. 되돌리기 시 Map에서 경로를 찾아 복원한다.

**캐시 무효화**: 같은 경로의 이미지를 복원하면 브라우저가 캐시된 "없는 이미지"를 보여준다. `?t=timestamp` 쿼리를 붙여 강제로 새로 로드한다.

## 커스텀 노드: fileAttachment

파일 첨부는 TipTap 기본 노드가 아니라 커스텀 블록 노드다. 마크다운에서는 `[filename](./.assets/filename)` 형태의 표준 링크로 저장되고, 에디터에서는 카드 형태로 렌더링된다.

**왕복(roundtrip) 문제**: @tiptap/markdown이 `.assets/` 링크를 일반 링크 마크로 파싱한다. `onCreate`에서 이 링크들을 감지해 fileAttachment 노드로 변환해야 하는데, 현재 감지 로직이 불완전하다 (WIP).

```
저장: fileAttachment 노드 → renderMarkdown → [file](./.assets/file)
로드: [file](./.assets/file) → @tiptap/markdown → link 마크 → (변환 필요) → fileAttachment 노드
```

## 코드블록 구문 강조

CodeBlockLowlight 확장 + lowlight 라이브러리로 ~40개 프로그래밍 언어를 자동 감지하고 색상을 입힌다. ReactNodeViewRenderer로 커스텀 UI를 씌워 언어 선택 드롭다운을 제공한다.

**즐겨찾기 언어**: 자주 쓰는 언어를 별표로 고정하면 드롭다운 상단에 표시된다 (localStorage 저장).

## 탭 시스템

문서 탭 외에 2종의 특수 탭이 있다:

| 탭 종류 | 아이콘 | 닫기 | 단축키 |
|---------|--------|------|--------|
| 검색 (tag-explorer) | Search | 불가 | Ctrl+Shift+F |
| 첨부파일 (attachment-explorer) | Paperclip | 불가 | Ctrl+Shift+A |
| 문서 (document) | - | 가능 | - |

특수 탭은 항상 탭바 왼쪽에 고정되어 있고, 구분선 뒤에 문서 탭들이 나온다. 문서 탭은 마우스 드래그로 순서를 바꿀 수 있다.

### 왜 HTML5 Drag API를 안 쓰나

Tauri WebView에서 HTML5 Drag API를 쓰면 금지 아이콘이 뜨면서 작동하지 않는다 (WebView 레벨에서 드래그 이벤트를 가로챔). 그래서 순수 mousedown/mousemove/mouseup 이벤트로 구현했다. 사이드바의 즐겨찾기 폴더 정렬, 파일 트리 드래그 이동도 같은 방식이다.

## 플로팅 툴바

이미지, 표, 첨부파일을 선택하면 해당 요소 위에 작은 툴바가 뜬다:

- **ImageToolbar**: 크기 조절 (원본/240/320/480/640/화면폭), 정렬 (왼쪽/가운데/오른쪽), 복사/잘라내기/삭제
- **TableToolbar**: 열/행 추가·삭제, 복사/잘라내기/삭제
- **FileToolbar**: 열기, 위치 열기, 다른 이름 저장, 복사/잘라내기/삭제

## 상태바

편집 영역 하단에 문서 정보와 설정을 한 줄로 보여준다:

```
[파일경로] [폴더열기] | [크기] | [줄수] | [글자수] | [태그들...] [태그입력]
                                                   → [고정폭▾] | [가운데▾] | [720px▾] | [숨기기]
```

오른쪽의 3개 드롭다운(폭 모드, 정렬, 페이지 폭)은 클릭하면 위로 팝업 메뉴가 나타나고, 기본값 항목에는 Home 아이콘이 표시된다.
